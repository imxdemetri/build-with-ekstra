Ekstra
Developer Platform

Build on the motion and presence layer of the real world.

Ekstra is a runtime, API, and SDK for products that read motion from a phone, webcam, or XR headset, and products that read the real world through public cameras. You pick a track, pull a starter, and ship.

Step 01 · Pick your track

Two ways to build on Ekstra.

Every app on Ekstra falls into one of two shapes. Motion in — something a person does becomes an action in your product. World out — you read a signal Ekstra derives from a public sensor. They share the same runtime, docs, and billing, but the starters, APIs, and consent model are different. Pick the one that matches your product.

Track A · Motion interaction

A hand raise becomes a click.

A person moves. Your app reacts. Phone IMU, webcam hand tracking, or XR pose — Ekstra turns raw sensor data into a normalized Motion Packet your code can subscribe to.

What it is
A hosted browser bridge (wss://ekstra.ai/ws) and a phone IMU ingest endpoint that, paired through a QR code, streams motion events into any web page. @ekstraai/controls-web is a companion metadata package — it describes the supported control profiles; the runtime wiring lives in each starter's vendored script.
What it's for
Replacing mouse, remote, or touchscreen in cases where those inputs are wrong — presentations from across the room, 3D inspection, accessibility navigation, kiosks, DOOH interactivity, VR/AR pointers.
How it fits
Sensor → phone-imu-provider (or camera/XR provider) → motiond → ws-bridge → your page via controls-web. You write event handlers, not drivers.
Ships today
  • starters/web-phone-pointer — phone IMU → on-screen pointer (Supported)
  • starters/presentation-remote — phone → deck navigation (Supported)
  • @ekstraai/controls-web@0.1.0 on npm — profile metadata, not a runtime
  • Hosted sandbox: ekstra.ai/build-with-ekstra/controller
  • Phone IMU ingest: POST ekstra.ai/api/phone-imu/ingest (live)
Consent model
  • User pairs their own device (QR scan, one tap).
  • Only derived Motion Packets leave the device — never raw video or sensor streams.

Track B · Camera observation

Find parking in Midtown right now.

Nobody moves. Your app reads the world. Ekstra ingests public DOT cameras, curb-rule data, meter occupancy, and router presence, and exposes them as a single queryable feed — so you don't have to stitch 50 state APIs together.

What it is
A REST API over 9,719 registered public cameras, 45,000+ NYC curb rules, live Seattle meter occupancy, and a growing pool of derived space signals (curb gaps, lot spaces, presence, traffic).
What it's for
Location intelligence without running the infrastructure. Parking apps, traffic dashboards, DOOH audience measurement, retail foot-traffic analytics, smart-city apps, research tools.
How it fits
Your client hits GET /api/v1/curb-rules/near, /api/v1/cameras, or a derived-signal endpoint. Ekstra handles ingestion, normalization, rate-limiting to upstream DOTs, and regulatory posture.
Ships today
  • starters/parking-map — curb rules on a Mapbox map in ~250 lines
  • GET /api/v1/curb-rules/near?lat=&lng=&radius_m= (live)
  • GET /api/v1/cameras — 9,719 cameras across 18 sources (live)
  • Full spec: /api/v1/openapi.json
Consent model
  • All feeds are public infrastructure; Ekstra redistributes derivatives only.
  • No personal data, no face recognition, no plate recognition. See regulatory notes in docs.

Still not sure? If your product needs a user to do something and your code reacts, you want Track A. If your product needs to read what's happening in a place and present that to a user, you want Track B. Apps that do both (e.g. a gym that measures form and tracks occupancy) run both tracks side by side on the same runtime.
Step 02 · What sits underneath

One runtime. One auth. One billing surface.

Both tracks run on the same Ekstra OS. You don't need to learn two systems — you just call different endpoints and pull a different starter.

Runtime

motiond (central daemon), ws-bridge (browser connectivity), phone-ingest, inference-sidecar, model-server. All running on Railway. You don't deploy any of it for the hosted sandbox.

Data layer

PostgreSQL + PostGIS for spatial queries, 27 ingestion adapters for public data sources (DOT cameras, curb rules, meter occupancy, weather, events, transit, roadwork), continuously refreshed.

Identity & billing

Motion Addresses (ekst1-prefixed ed25519 keypairs) identify devices. Developer API keys (eko_…) identify apps. A revenue share for published apps is the design target (80 / 15 / 5 — developer / Ekstra / stakers) — there is no live payout flow today.

Step 03 · Motion interaction, in depth

If you picked Track A.

Control profiles are the contract between a motion source and your app. Pick one that matches what you're building; the runtime handles the mapping from raw sensor data to your handlers.

Minimal integration

Clone starters/web-phone-pointer, open the HTML, point it at the hosted sandbox, scan the QR with a phone. The vendored ekstra-motion-web.js in the starter is the runtime; @ekstraai/controls-web gives you typed profile metadata if you need it. No backend to deploy.

Step 04 · Camera observation, in depth

If you picked Track B.

The camera observation track is a straight REST API with no auth required for public reads. Rate limits apply; production keys unlock higher throughput and derived signals.

Endpoints available

  • GET /api/v1/curb-rules/near — 45k+ NYC rules, time-aware
  • GET /api/v1/cameras — 9,719 public DOT cameras, paginated
  • Each camera record carries image_url for direct upstream fetch
  • GET /api/v1/openapi.json — full machine-readable spec

Coverage today

NYC (all 5 boroughs, live), LA / SF / San Diego / Sacramento, Seattle (+ live meter occupancy), Portland, Chicago, Philadelphia, Austin, Las Vegas, Louisville, and 8 more metros. Full list in the camera coverage doc.

What you can't do

No face recognition, no license-plate reads, no raw-frame redistribution. Ekstra does derivative analysis only — the space object (lat/lng, occupancy, confidence) is what leaves the system. This is enforced in the platform, not a policy.
